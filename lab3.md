# 2е задание

### Backend: Java (Spring Boot) & Spring

**Java (Spring Boot)** — это промышленный стандарт для создания сложных, высоконагруженных и надежных бэкенд-систем.

*   **Обоснование:** Spring Boot предоставляет готовую к использованию, конфигурируемую платформу, что резко ускоряет разработку. Он следует принципам инверсии контроля и Dependency Injection, что делает код модульным, легко тестируемым и сопровождаемым.
*   **Удобство:** Автоконфигурация избавляет от написания шаблонного кода. Встроенные серверы (Tomcat) и обширная документация упрощают запуск и развитие проекта.

**Spring MVC** — стандартный фреймворк для построения веб-приложений и REST API.

*   **Обоснование:** Четкое разделение на Model, View, Controller, что обеспечивает чистую архитектуру и предсказуемость кода.
*   **Удобство:** Простая аннотационная конфигурация (`@Controller`, `@RequestMapping`), интегрированная валидация и обработка исключений.

**Spring Data JPA** — абстракция для работы с базами данных.

*   **Обоснование:** Резко сокращает объем шаблонного кода для CRUD-операций. Позволяет оперировать объектами Java, а не сырыми SQL-запросами.
*   **Удобство:** Достаточно объявить интерфейс-репозиторий, и Spring автоматически реализует основные методы (`save`, `findById`, `findAll`). Поддержка написания кастомных запросов через `@Query`.

**WebSocket (STOMP)** — протокол для двусторонней реального времени связи между клиентом и сервером.

*   **Обоснование:** Идеален для функций, требующих мгновенного обновления: чаты, уведомления, онлайн-игры, совместное редактирование. STOMP — это надстройка, которая добавляет понятную семантику сообщений (как в Kafka/RabbitMQ).
*   **Удобство:** Spring предоставляет прозрачную интеграцию STOMP поверх WebSocket, что упрощает реализацию pub/sub модели на стороне сервера и клиента.

**Spring Security** — стандарт де-факто для аутентификации и авторизации.

*   **Обоснование:** Мощный и гибкий фреймворк, закрывающий все основные векторы атак и сценарии безопасности (OAuth2, JWT, Basic Auth, ролевая модель).
*   **Удобство:** Глубокая интеграция с Spring MVC и Servlet API. Настраивается через конфигурацию, а не код, для стандартных сценариев.

**Kafka** — распределенная платформа потоковой обработки событий.

*   **Обоснование:** Используется для построения асинхронных, отказоустойчивых и масштабируемых систем. Обеспечивает надежную доставку сообщений между микросервисами или частями системы. Идеальна для обработки пиковых нагрузок.
*   **Удобство:** Spring Kafka предоставляет удобные шаблоны для отправки и получения сообщений, абстрагируя разработчика от низкоуровневого API.

**Redis** — хранилище «ключ-значение» в оперативной памяти.

*   **Обоснование:** Используется как высокопроизводительный кэш для снижения нагрузки на основную БД и для хранения временных данных (например, сессий). Также часто выступает в роли брокера сообщений для простых сценариев.
*   **Удобность:** Spring Data Redis предлагает знакомый шаблон репозитория и шаблоны для простой работы с данными.

**Spring Cloud** — набор инструментов для построения распределенных систем (микросервисов).

*   **Обоснование:** Предоставляет готовые реализации для сервис-дискавери (Eureka), конфигурации (Config Server), балансировки нагрузки (Ribbon) и т.д. Позволяет собрать надежный микросервисный архитектурный стиль.
*   **Удобство:** Интегрируется с экосистемой Spring Boot, позволяя быстро "поднять" инфраструктурные сервисы.

*   **Обоснование:** Это полнофункциональная, объектно-реляционная СУБД с открытым исходным кодом. Поддерживает сложные запросы, транзакции (ACID), JSON-типы данных, полнотекстовый поиск. Надежна и хорошо известна в индустрии.
*   **Удобство:** Отличная документация, большое сообщество. Легко развертывается и администрируется. Совместима со стандартом SQL.

*   **Обоснование:** Позволяет генерировать HTML на стороне сервера. Хорош для SEO, быстрого старта и приложений, где не требуется сложная интерактивность на клиенте. Естественно интегрируется с Spring MVC.

*   **Обоснование:** Обеспечивают богатый, интерактивный пользовательский опыт, сравнимый с нативными приложениями. Весь UI рендерится на клиенте, а с сервером взаимодействие идет через REST/GraphQL API.
*   **Удобство:** Компонентный подход, который упрощает разработку и поддержку сложных интерфейсов. Огромные экосистемы библиотек и инструментов.

# 3 задание

### Общая архитектура системы

Система построена по микросервисной архитектуре с выделенными сервисами для различных бизнес-доменов. Взаимодействие между сервисами осуществляется асинхронно через Kafka и синхронно через REST API с использованием Service Discovery. Единой точкой входа является API Gateway.

---

### Декомпозиция модулей (микросервисов)

#### 1. **Discovery Server (Eureka)**
**Назначение:** Централизованный сервис для регистрации и обнаружения микросервисов.
*   **Функции:**
    *   Регистрация экземпляров всех микросервисов.
    *   Предоставление клиентам (Gateway, другим сервисам) актуального списка доступных экземпляров каждого сервиса.
    *   Отслеживание состояния работоспособности зарегистрированных сервисов (heartbeat).
*   **Взаимодействие:** Все остальные сервисы являются его клиентами.

#### 2. **API Gateway**
**Назначение:** Единая точка входа для всех клиентских запросов, отвечающая за маршрутизацию, аутентификацию и кросс--cutting concerns.
*   **Функции:**
    *   **Маршрутизация:** Перенаправление входящих запросов (например, `/api/users/**` -> `user-service`) на соответствующие бэкенд-сервисы.
    *   **Аутентификация:** Проверка JWT-токена в заголовке запроса. Незащищенные маршруты (логин, регистрация) пропускаются без проверки.
    *   **Авторизация:** Проверка прав доступа к определенным эндпоинтам (например, роль `ADMIN` для доступа к `/admin/**`).
    *   **Логирование:** Централизованное логирование входящих запросов и ответов.
    *   **Rate Limiting:** Ограничение количества запросов для защиты от злоупотреблений.
    *   **Упрощение клиента:** Клиент (Web, Mobile) взаимодействует только с одним доменом Gateway.
*   **Взаимодействие:** Обращается к Discovery Server для получения адресов сервисов. Взаимодействует с `user-service` для валидации учетных данных при аутентификации.

#### 3. **User Service**
**Назначение:** Управление данными и жизненным циклом пользователей.
*   **Функции:**
    *   Регистрация нового пользователя.
    *   Аутентификация (логин) и выдача JWT-токена.
    *   Управление профилем пользователя (просмотр, редактирование).
    *   Управление ролями и правами пользователей (например, `USER`, `ADMIN`).
*   **Взаимодействие:**
    *   Синхронное: Принимает запросы через Gateway.
    *   Публикует события в Kafka (например, `USER_REGISTERED`, `USER_PROFILE_UPDATED`) для уведомления других сервисов.

#### 4. **Order Service**
**Назначение:** Полное управление заказами.
*   **Функции:**
    *   Создание нового заказа.
    *   Просмотр списка заказов и деталей конкретного заказа.
    *   Изменение статуса заказа (например, `CREATED` -> `PAID` -> `SHIPPED` -> `DELIVERED`).
    *   Управление составом заказа (товары/услуги).
*   **Взаимодействие:**
    *   Синхронное: Получает запросы через Gateway. Может делать синхронные вызовы к `user-service` (через Discovery) для проверки существования пользователя.
    *   Асинхронное: Публикует события в Kafka (например, `ORDER_CREATED`, `ORDER_STATUS_CHANGED`), на которые подписан `notification-service`.

#### 5. **Notification Service**
**Назначение:** Отправка уведомлений пользователям через различные каналы.
*   **Функции:**
    *   **Подписка на события:** Прослушивание соответствующих топиков Kafka.
    *   **Шаблоны уведомлений:** Формирование текста уведомлений на основе полученных событий и шаблонов.
    *   **Отправка по Email:** Интеграция с SMTP-сервером или внешним email-провайдером (SendGrid, Mailgun).
    *   **Отправка в Telegram:** Интеграция с Telegram Bot API для отправки сообщений.
    *   **Управление настройками уведомлений:** (Опционально) Учет предпочтений пользователя (какие уведомления и куда получать).
*   **Взаимодействие:**
    *   Асинхронное: Получает события от `user-service` и `order-service` через Kafka. Не имеет REST API для прямых вызовов, что делает его полностью асинхронным и слабосвязанным.

---

### Схема данных и взаимодействия

1.  **Клиент** делает запрос на `API Gateway`.
2.  **Gateway** проверяет аутентификацию (для защищенных маршрутов) и находит адрес целевого сервиса через **Discovery Server**.
3.  **Gateway** перенаправляет запрос на нужный сервис (например, `user-service` или `order-service`).
4.  **Order Service** при создании заказа публикует событие `ORDER_CREATED` в **Kafka**.
5.  **Notification Service**, подписанный на этот топик, получает событие, формирует уведомление и отправляет его пользователю по Email и/или в Telegram.
6.  **User Service** при регистрации публикует событие `USER_REGISTERED`, на которое также реагирует **Notification Service**, отправляя приветственное письмо.

### Преимущества данной декомпозиции

*   **Высокая связность, слабое зацепление:** Каждый сервис отвечает за свою строго определенную бизнес-область.
*   **Масштабируемость:** Можно независимо масштабировать каждый сервис. Например, `notification-service` можно легко увеличить при высокой нагрузке на рассылку.
*   **Отказоустойчивость:** Если `notification-service` временно недоступен, сообщения будут накапливаться в Kafka и будут обработаны после его восстановления, не затрагивая работу `order-service`.
*   **Технологическая гибкость:** Каждый сервис может быть разработан и развернут независимо (в рамках выбранного стека).
